# Core Workflow & Lifecycle

SDD is a structured document-driven workflow for complex feature development that combines markdown artifacts, phase detection, script automation, and supervision hooks. This document explains how the core lifecycle works, how to handle complex multi-subsystem features with work packages, and how to manage explicit feature selection when multiple features are in flight.

## Supervised Workflow: The Agent Harness (0001)

SDD graduates from a document convention to an active agent harness through a three-layer hook architecture that enforces correctness, maintains continuity, and detects drift.

### Three-Layer Hook Design

**Command hooks** (every tool use): Deterministic, fast checks that run on every Edit, Write, and Bash command. These read machine-readable sidecar files (scope.txt, state.json) for scope-drift detection, state preservation, and completion gating. When you edit a file outside the defined feature scope during the implement phase, a warning appears without blocking the edit.

**Prompt hooks** (at step boundaries): Semantic analysis powered by Haiku that triggers at critical moments like task completion. These read raw artifacts (tasks.md, research.md) plus the actual git diff to assess whether your work stayed in scope and meets quality expectations. The review is informational — Claude sees the assessment and acts on it.

**Agent hooks** (rare deep verification): Full artifact inspection when a feature reaches the close phase. These verify that all requirements were addressed and that the artifact chain is complete.

### Session Continuity via Feature-Scoped Context

When you resume a session, the SessionStart hook injects structured state from three feature-scoped context files (the "three-tier context"):

- **continuity.md**: Session state ledger. Before context compaction, a PreCompact hook appends a "Compaction Snapshot" with timestamp, current phase, active task, and key scratchpad observations. On resume, the full execution state is injected, so Claude knows exactly where it was.

- **scratchpad.md**: Working memory for the feature. Captures observations, blockers, design decisions, and interim results. A rotating nudge (every other prompt) encourages you to keep it fresh.

- **knowledge.md**: Durable facts, design patterns, and gotchas discovered during the work. At the close phase, entries can graduate to the project-level KNOWLEDGE.md for long-term retention.

All three files live in `.features/<feature_path>/` alongside your brief, spec, research, and other artifacts, creating a self-contained feature workspace.

## Feature Directory Layout

The default layout remains flat:

```text
.features/0001-user-auth/
```

For larger programs, a feature may be placed under an explicit group path:

```text
.features/platform/search/0008-index-refresh/
```

The feature identity is still the basename (`0008-index-refresh`). The feature path is the storage path relative to `.features` (`platform/search/0008-index-refresh`). Commands accept exact nested paths, and unambiguous basename or numeric-prefix selectors still work. If a basename or prefix matches multiple nested features, resolution fails as ambiguous instead of guessing. Unsafe selectors such as absolute paths, traversal, empty path segments, hidden segments, and `wp-*` paths are rejected.

Create grouped features explicitly:

```bash
sdd init index-refresh --group platform/search "Refresh search index lifecycle"
```

Group folders are organizational containers, not features. Discovery is marker-based: a directory becomes a feature only when it has a numeric feature basename and feature artifacts or sidecars. Generated sidecars such as `summary.md`, `summary-compact.md`, `changes.log`, and `customization.yaml` are ignored for both flat and nested features by the setup-managed `.gitignore` patterns.

### Machine-Readable Sidecars

The harness follows the principle: "Make the writer produce a simple format, not the reader parse a complex one." Markdown artifacts remain the source of truth; hooks read simple sidecar files:

- **scope.txt**: One in-scope file path per line, generated by `sdd research` from the Affected Files table. Command hooks grep this file to detect scope drift on every edit.

- **state.json**: Stable feature-local cache derived from `sdd status --json`. Updated asynchronously after every tool use, hooks read it for feature state without re-running detection. Invocation-scoped metadata from a particular CLI call is not persisted here.

- **changes.log**: Append-only log of every file modification (timestamp | tool_name | file_path). Used by task completion hooks for scope analysis and by the review phase for traceability.

## Complex Features: Work Packages (0002)

For features spanning multiple independent subsystems (like a full harness with command hooks, prompt hooks, and agent hooks), a flat tasks.md with 20+ items becomes hard to navigate and loses its utility as a progress tracker. Work packages let you decompose a complex feature into semi-independent units, each with its own mini-SDD lifecycle.

### When to Use Work Packages

You don't need work packages for small-to-medium features. Use them when:
- A feature has 15+ tasks spanning multiple subsystems
- Different subsystems could benefit from separate reviews or hand-offs
- You want clean stopping points (pause after completing a work package)
- Independent subsystems could run in parallel via subagents

### How Work Packages Work

Create optional `wp-NN-name/` subdirectories within your resolved feature folder. By convention, the presence of any direct-child `wp-*/` directory activates WP mode — no flags, no config. Work packages are never discovered as features. Each work package inherits the feature-level brief and spec, but has its own:

- **tasks.md**: List of tasks specific to this work package (T-01, T-02, etc., scoped locally)
- **review.md**: WP-level review with verdict (pass/fail). Feature-level review only becomes creatable when all WP reviews pass.
- **scratchpad.md** (optional): WP-specific working memory
- **research.md, plan.md** (optional): WP-level detail when complexity warrants

Example structure:

```
.features/0001-sdd-agent-harness/
  brief.md
  spec.md
  research.md
  plan.md
  scratchpad.md
  continuity.md
  knowledge.md
  wp-01-command-hooks/
    tasks.md          (T-01, T-02, ...)
    review.md
    scratchpad.md
  wp-02-prompt-hooks/
    tasks.md
    review.md
  wp-03-documentation/
    tasks.md
    review.md
```

In normal WP lifecycle the parent feature may omit root `tasks.md`. The WP task files are the execution contract; root `tasks.md` is not created as a stub during normal close/check paths.

### Active WP Detection

When a feature has work packages, `sdd-phase.sh` and `sdd task` commands automatically detect the **active work package**:

1. First WP with in-progress tasks (any task in the implement phase)
2. Else, first WP with pending tasks (not yet started)
3. Used to scope all `sdd task` commands to that WP only

Phase reporting distinguishes the parent artifact phase from the WP execution state. `raw_phase` is based on feature-level artifacts, while `effective_phase` / `execution_phase` are WP-aware and drive hook guidance, phase-skill routing, stop checks, and status summaries.

You always work on one WP at a time. When you complete its review, that's a natural stopping point — the stop gate allows it, and you can pause confidently knowing which WP is done.

### Work Package Status and Progress

`sdd status <feature>` shows per-WP breakdown:

```
FEATURE 0001-sdd-agent-harness (WP mode)
├─ WP-01 command-hooks        research  [████░░░░░] 4/10 tasks
├─ WP-02 prompt-hooks         pending   [░░░░░░░░░░] 0/8 tasks
├─ WP-03 documentation        pending   [░░░░░░░░░░] 0/5 tasks
---
TOTAL: 3 WPs (4/23 tasks)
```

Feature-level progress aggregates across all WPs. Hook output includes WP summary: total count, done/active/pending breakdown, and the active WP name + phase.

`sdd status --json` and feature-local `state.json` expose this same aggregate state through `task_source.mode=work_packages`, per-WP task summaries, WP-attributed `current_task.display_ref`, and the WP-aware effective phase / execution phase. Consumers that need the parent artifact phase should read `raw_phase`.

### Feature-Level Review with WP Rollup

When all individual WP reviews pass, the feature-level review becomes available. It includes a rollup table showing each WP's verdict, allowing you to verify the feature as a coherent whole before marking it closed.

Close readiness follows the same order: all WP tasks must be complete, every WP review must pass, then the feature-level review must pass. A complete WP feature closes normally without root `tasks.md`.

## Feature Selection: Explicit vs. Auto-Detect (0006)

When you have multiple features in flight (a common scenario during lab work), SDD's active feature detection uses heuristics: in-progress tasks take priority, then pending tasks, then the first non-closed feature. But heuristics can guess wrong — especially when you create a new feature or have stale unclosed features.

### Explicit Selection with `sdd select`

Pin the active feature explicitly:

```bash
sdd select 0006-sdd-feature-selection   # Pin by full name
sdd select 0006                          # Pin by numeric prefix (if unambiguous)
sdd select                               # Show current selection and auto-detect fallback
sdd select --clear                       # Unpin and return to auto-detect
```

When you run `sdd select <feature>`, it writes the canonical feature path to `.features/.active`; for flat features this is the same value as the legacy feature ID. All hooks, scripts, and subagents check this file first before falling back to auto-detect. If `.active` points to a closed or non-existent feature, it falls through silently to auto-detect.

### Auto-Selection on Feature Creation and Close

- **`sdd init <feature>`** automatically writes the new flat feature ID or grouped feature path to `.features/.active`, selecting it immediately so your first prompt already has the right context.
- **`sdd close <feature>`** clears `.features/.active` if it matches either the closed feature ID or path, preventing stale selection.

### Source Indicator in Hook Output

Hook output includes a `[selected]` tag when the active feature was explicitly pinned:

```
>>> SDD [selected] 0006-sdd-feature-selection | phase: implement | task: T-05 | 2/8 done
```

Without the `[selected]` tag, the feature was auto-detected — useful for knowing whether to trust the context or check the feature manually.

### Feature Index (0031)

SDD maintains a persistent `.features/INDEX.md` at the root of the features directory — a human-readable overview table and per-feature sections listing every feature with its number, name, phase, status, and description. The index provides a single entry point for understanding the entire feature landscape without scanning individual directories.

#### INDEX.md Structure

The INDEX.md file contains three sections:

1. **Summary stats**: Total feature count, open/closed breakdown, and generation timestamp
2. **Overview table**: Columns for feature number, feature name, current phase, task progress (completed/total), and last updated date. One row per feature.
3. **Per-feature detail sections**: For each feature, a titled subsection with phase, created date, task progress bar, and the feature's description

#### Description Source Priority Chain

Feature descriptions are sourced using a three-level fallback chain:

1. If the feature has a `summary-compact.md` file with a "Goal:" line, use the first line of that (truncated at 200 characters)
2. Else, use the title from the feature's `brief.md` YAML frontmatter
3. Else, use the feature directory name (basename)

This ensures every feature has a meaningful description without requiring additional fields.

#### Index Commands

```bash
sdd index              # Display INDEX.md (rebuild first if missing)
sdd index rebuild      # Regenerate INDEX.md from disk state
sdd index validate     # Check consistency between INDEX.md and disk
sdd index --json       # Machine-readable JSON output
```

#### Automatic Rebuild Triggers

The index auto-rebuilds automatically on lifecycle events that change the feature set or phase:

- `sdd init <feature>`: Rebuilds after creating a new feature
- `sdd close <feature>`: Rebuilds after closing a feature
- `sdd reopen <feature>`: Rebuilds after reopening a closed feature
- `sdd summary <feature>`: Rebuilds after generating a feature summary

In all cases, the rebuild is non-interactive and included as a final step of the command. The `--quiet` flag suppresses rebuild progress output.

## Workflow Phases and State

Every SDD feature progresses through phases: brief → research → plan → implement → review → close. Phase detection is automatic based on artifact state (which files exist, whether tasks are pending or done, whether review passed). The harness enforces transitions via the stop gate.

### Stop Gate Enforcement

The sdd-stop-gate.sh hook blocks premature stopping:
- During implement/review phases, allows stopping only when the current task/phase is complete
- Blocks stopping mid-task (pending or in-progress tasks remain)
- For work package features: allows stopping at WP boundaries (active WP review passed, next WP not started) but blocks mid-WP

When blocked, the stop gate outputs which tasks must be completed or reviewed before you can pause.

## Integration: How It All Works Together

**Session start:** sdd-hook.sh checks `.features/.active` to determine the active feature. If present, it's used; otherwise, auto-detect runs. The SessionStart hook injects continuity.md if it exists (rich context from the previous session including phase, task, and observations).

**During implementation:** Every edit or file write is checked by the PreToolUse scope-drift hook against scope.txt. Warnings appear if you edit files outside the defined scope. After every tool use, changes.log is updated for traceability.

**Rubric advisor hooks:** Around artifact authoring (Edit/Write to `.features/*/spec.md`, `brief.md`, `research.md`, `plan.md`, `review.md`) and ahead of phase boundary commands (`sdd spec`, `sdd research`, `sdd plan create`, `sdd close`), a pre-boundary advisor hook fires to surface rubric guidance, and a post-authoring advisor hook fires to assess the produced artifact. Both are informational signals, not blocking gates.

**Task completion:** `sdd task complete <feature> T-xxx` runs the task's verification command (if present, with a 30-second timeout). If verification passes, the hook triggers a Haiku-powered prompt hook that assesses whether the work stayed in scope. The task is marked complete.

#### Task Dependencies

Tasks may declare dependencies on other tasks using the `Depends-on:` field in their task block. The plugin validates the dependency graph, enforces correct execution order, and auto-generates task-scoped file tracking.

##### Dependency Field Syntax

The `Depends-on:` field supports three equivalent formats for backward compatibility:

- Legacy CSV: `Depends-on: T-001, T-002`
- YAML-flow: `Depends-on: [T-001, T-002]`
- Canonical JSON array: `Depends-on: ["T-001", "T-002"]`

New artifacts and templates always emit canonical JSON array syntax. The parser normalizes all three formats to the same internal representation.

##### Task Selection: sdd task next

`sdd task next` filters pending tasks to only those whose declared dependencies are complete (status: completed). It then returns the first dependency-ready task in document order. If no dependency-ready task exists but pending tasks remain, it reports the blocking chain — which tasks must be completed first. This ensures the agent always works on the correct prerequisite task.

For WP features, `sdd task list`, `next`, `start`, and `complete` operate on the active WP by default. Use `sdd task list --all <feature>` when you need a cross-WP inventory; it lists every WP task with `WP-NN/T-xxx` attribution so repeated local task IDs are unambiguous.

##### Task Start with Dependency Checking

`sdd task start` validates all dependencies before starting a task:

- If all `Depends-on` tasks are `completed`, the task starts normally
- If any dependency is pending or in-progress, `sdd task start` exits with code 4 and lists which dependencies are incomplete
- Pass `--force` flag to override dependency checks and start the task anyway (for exceptional cases)

##### Task-Scope Sidecar Generation

When `sdd task start` begins a task that has a `Files:` field, it automatically writes a `.features/<feature>/task-scope.txt` file containing exactly the file paths listed in that task's Files field — one path per line. This file is used by the PreToolUse scope-drift detection hook to enforce tighter scope boundaries during implementation. If legitimate scope discovery changes the in-progress task's `Files:` field, run `sdd task scope-refresh <feature> <T-xxx>` to rebuild `task-scope.txt` without resetting task status. When the task is completed or reset, the `task-scope.txt` file is removed. If multiple tasks are pending, the most recently started task's scope is active.

##### Dependency Graph Inspection

`sdd task graph <feature>` returns a JSON object describing the full dependency graph:

- `nodes`: Array of all tasks with their IDs, status (pending/in-progress/completed/blocked), and titles
- `edges`: Array of dependency pairs `[source_task, target_task]` representing "source depends on target"
- `ready`: Array of task IDs that are currently dependency-ready (pending and all dependencies completed)
- `problems`: Object containing `cycles`, `missing_ids`, and `self_dependencies` if any are found

##### Dependency Validation

`sdd check tasks` validates the dependency graph for structural errors and exits with code 4 if any are found:

- **Cycles**: Detects circular dependencies (e.g., T-001 → T-002 → T-001)
- **Missing IDs**: Reports task IDs in `Depends-on` fields that don't exist as actual task headings
- **Self-references**: Detects tasks that depend on themselves (T-001 depends on T-001)
- **Work package cross-refs**: Validates WP-cross-reference syntax (e.g., WP-02/T-005)

For each error, the output identifies the task, the problematic field, and what's wrong (e.g., "T-007 depends on T-999 which does not exist").

**Stopping:** The Stop hook reads the current effective phase / execution phase and task state (via sdd-phase.sh and the active WP if present). If tasks remain pending, review isn't done, or (in WP mode) the active WP isn't finished, stopping is blocked with guidance on what to complete.

**Session resume:** PreCompact hook appends a compaction snapshot to continuity.md before context is lost. On the next SessionStart, this full state is injected, letting Claude resume exactly where it left off.

**Feature close:** `sdd close <feature>` checks all phases, clears `.features/.active` if it matches, and optionally promotes feature-scoped knowledge.md entries to the project KNOWLEDGE.md. In WP mode, close evaluates aggregate WP task/review readiness before requiring the final feature-level review.

#### Archive Docs

After closing a feature, run `sdd archive <feature>` to auto-generate two human-readable documents inside the feature folder:

- `README.md` — 500-800 word narrative synthesis covering the problem, approach, decisions, and outcomes.
- `DOCS.md` — technical reference for the implemented solution.

These documents are dispatched to the background `sdd-archivist` subagent and make closed feature history accessible without reading raw artifacts.

### Auto-Setup: Bootstrapping SDD Into a Project (0032)

`sdd setup` installs the full SDD standalone configuration into the current working directory, making SDD independent of the plugin:

```bash
sdd setup             # install agents, skill, hooks, CLAUDE.md block, .features/
sdd setup --dry-run   # preview changes without writing
sdd setup --force     # skip backup prompt but still back up modified files
sdd setup --json      # machine-readable output
```

The installed copy works completely standalone — the plugin does not need to remain loaded after setup. Re-running `sdd setup` after a plugin update safely upgrades all components while preserving user customizations.

#### Installed Files and Directories

`sdd setup` writes the following to the target project:

**Agents** (`.claude/agents/`)
- Copies all 6 SDD agent definitions from the plugin to `$PROJECT/.claude/agents/sdd-*.md`

**Skill** (`.claude/skills/spec-driven-development/`)
- Copies the full skill directory structure: SKILL.md, scripts/, templates/, references/, docs/
- Does NOT copy agents/ (handled separately) or evals/ (development-only)

**CLI entry point** (`.claude/bin/sdd`)
- Creates a shell wrapper that invokes the local skill copy
- Has path resolution that works independently of the plugin

**Hook registrations** (`.claude/settings.json`)
- Registers all hooks from the plugin's hooks.json into the project's settings.json
- Replaces `${CLAUDE_PLUGIN_ROOT}` path variables with `${CLAUDE_PROJECT_DIR}/.claude`
- Preserves existing hooks from other tools

**Project instructions** (`CLAUDE.md`)
- Creates or updates CLAUDE.md with an SDD instructions block
- Uses marker comments (`<!-- sdd-setup:begin -->` and `<!-- sdd-setup:end -->`) for idempotent updates
- Appends to existing content if CLAUDE.md already exists

**Features directory** (`.features/`)
- Creates if it does not exist (empty state for new projects)

**Version manifest** (`.sdd-version` and `.sdd-manifest.json`)
- `.sdd-version`: Simple version string for idempotency checks
- `.sdd-manifest.json`: SHA-256 hashes of all installed files, version, and installation timestamp

#### Backup Strategy

When re-running `sdd setup` with an updated plugin:

- Existing files are checked against `.sdd-manifest.json` to detect user modifications
- If a file has been modified by the user (hash mismatch), it is backed up to `.sdd-local-patches/<timestamp>/` before overwriting
- The `--force` flag skips the interactive confirmation prompt but still performs backups (no data loss)
- On each setup run, old `sdd-*.md` agent files in `.claude/agents/` are removed before copying new versions (clean slate per version)

#### Idempotency and Version Checking

Re-running `sdd setup` multiple times is safe:

- Setup checks `.sdd-version` to determine if components are already current
- If all installed files match the current plugin version (by hash), setup reports "no changes needed" and exits cleanly
- Only files that differ are overwritten, reducing unnecessary writes and backups
- New features in the plugin are pulled in on re-run

#### Validation After Setup

After installation completes, `sdd setup` runs `sdd check` to verify the installation is functional:

- Verifies that hooks are correctly registered in `.claude/settings.json`
- Checks that the CLI entry point is executable
- Validates that the skill directory structure is complete
- Exits with non-zero status and reports errors if anything is misconfigured

This ensures that the installed SDD is ready to use immediately after setup completes.

## Project Customization

SDD supports a thin per-project customization layer that injects user-authored guidance into the plugin's existing emission points (SessionStart context, phase nudges, subagent dispatch packets, and continuity recovery). Customization is append-only — it never replaces the plugin's static text — and is invisible (zero overhead) for projects that don't opt in.

### Customization Files

- **`.sdd/customization.yaml`** — project-level scope, committed to git. Holds shared conventions, tech-stack reminders, and durable rules that every contributor's session should see.
- **`.features/<id>/customization.yaml`** — optional feature-level companion, gitignored by default. Holds feature-specific guidance and discoveries that should not bleed into other features.

Both files share a single schema: top-level `schema_version: 1` and a `notes` dict whose values are lists of short string bullets. Users edit the YAML directly with any text editor; Claude can also append to it during a session.

### Scope Keys

The `notes` dict accepts 15 known scope keys, grouped by purpose:

| Group | Keys |
| --- | --- |
| Always-on | `global`, `learned` |
| Phases (7) | `brief`, `design`, `research`, `plan`, `implement`, `review`, `close` |
| Subagent roles (6) | `research-scout`, `plan-drafter`, `implementer`, `spec-reviewer`, `quality-reviewer`, `archivist` |

Unknown keys are reported as non-fatal warnings by the validator but do not block reads.

### Interpolation Tokens

Bullet text is substituted at injection time using exactly four tokens:

- `{{FEATURE}}` — active feature ID (e.g., `0045-sdd-project-customization`)
- `{{FEATURE_ID}}` — numeric prefix only (e.g., `0045`)
- `{{PHASE}}` — active phase name (e.g., `research`)
- `{{DATE}}` — current UTC date in `YYYY-MM-DD`

Unknown `{{...}}` tokens are left literal in the output and warned on by the validator.

### Injection Model

Hooks read the YAML files and inject only the scope slices relevant to the current context:

- **SessionStart**: `notes.global` and `notes.learned` always; `notes.<phase>` only when the active feature is in that phase. Output appears as a new sibling `<project_customization>` block alongside the existing `<sdd_state>` and `<sdd_directives>` blocks — never replacing them.
- **SubagentStart (dispatch packets)**: `notes.global`, `notes.<agent_role>`, and `notes.learned`. This is the only mechanism by which subagents see project context, since they start with empty conversation context.
- **Continuity recovery**: `notes.learned` is folded into the `continuity.md` snapshot mechanism so PreCompact/PostCompact recovery preserves discoveries made mid-feature.

When both project and feature files are present, matching scope slices are merged with feature-level bullets appended after project-level bullets, and each contributing source is visibly labeled (e.g., `[project/research]` and `[feature 0045/research]`).

### Commands

- **`sdd customize init`** — scaffolds an empty `.sdd/customization.yaml` with all 15 known scope keys present as empty lists and inline comments explaining each key's role.
- **`sdd customize check <file>`** — validates schema conformance: top-level keys, value shapes, known scope keys, per-bullet length ≤ 280 chars, per-key bullet count ≤ 50, total file size ≤ 8 KB. Non-zero exit on hard failures; warnings on soft issues.

See `references/customization.md` for the full schema reference, examples, and validation rules.

## Key Conventions

- **Task IDs**: Flat features use T-01, T-02, etc. Work package features scope task IDs locally within each WP (wp-01/T-01, wp-02/T-01), with cross-WP references using WP-NN/T-xx notation.
- **Graceful degradation**: All hooks exit cleanly (exit 0) if no active feature exists, jq is missing, or scope.txt is absent. Hooks are never required to succeed — they're supervision, not blocking gates.
- **Artifacts as source of truth**: scope.txt, state.json, and changes.log are caches for fast reading. Markdown artifacts (research.md, tasks.md, etc.) remain the source of truth. Hooks read caches, Claude edits markdown.
- **Feature workspace scope**: All feature-specific context (continuity.md, scratchpad.md, knowledge.md, scope.txt, state.json, changes.log) lives in `.features/<feature>/` alongside brief, spec, research, and other artifacts.
